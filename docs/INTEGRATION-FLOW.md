# Integration Flow — Frontend ↔ API

End-to-end trace of how a user action travels from a button click in the browser to a row in Postgres and back to the screen, plus the conventions that keep all features following the same path.

**Scope:** Phase 1b. The backend (Phase 1a) is already shipped — see [`API-ROADMAP.md`](API-ROADMAP.md) and [`API-ARCHITECTURE.md`](API-ARCHITECTURE.md).

---

## 1. The canonical request path

```
[User] → [Page (RSC)] → [Hook (RQ)] → [SDK] → [Next.js Route] → [Use Case] → [Repository] → [Prisma] → [Postgres]
                                                                   │
                                                            audit_log emit
                                                                   │
                                                            (Task 14 timeline,
                                                             Task 16 dispatcher)
```

Every feature touches the same boundaries. Each boundary has a single rule:

| Boundary | Rule |
|---|---|
| Page (RSC) | Pre-fetch via `prefetchQuery`; hydrate to RQ cache; pass nothing but `id`s to children |
| Hook | Wraps RQ; encodes the query key; chooses staleness; never imports SDK in components |
| SDK | Generated by `@graphql-codegen` from `*.graphql` files OR thin REST wrapper; both go through `~/lib/api-client` |
| Route | Thin: parse params + Zod-parse body → use case → map result → response |
| Use case | Business logic; receives repos as params; returns discriminated union |
| Repository | Prisma access only; selects + indexes |
| Audit log | Write at the use-case layer; never in routes or repos |

## 2. Authentication flow

### 2.1 Login

```
[User submits form]
  ↓
[useLogin mutation] → POST /api/auth/login (cookie: none)
  ↓
[loginUseCase] checks credentials, signs JWT
  ↓
[Route] sets httpOnly cookie `session`, returns 200 + user payload
  ↓
[useLogin onSuccess] → router.push('/dashboard')
  ↓
[Dashboard page (RSC)] reads `session` cookie via `getPrincipal(req)`,
                       fetches initial data server-side, hydrates RQ
```

### 2.2 Subsequent authed requests

- Browser auto-sends the `session` cookie (`SameSite=Lax`).
- Both REST and GraphQL routes call `await getPrincipal(req)` to extract the user.
- Server components reach the same helper via `headers()`.

### 2.3 Logout / 401 handling

- 401 from any hook → global RQ `onError` handler clears the cache and routes to `/login`.
- A small auth observer in `app/(app)/layout.tsx` watches for 401s and triggers the same redirect.

## 3. Data fetching strategy (per screen type)

### 3.1 List + detail (read-heavy)

| Path | Tool |
|---|---|
| Server pre-fetch | RSC `await fetch` or RQ `prefetchQuery` |
| Client cache | `<HydrationBoundary>` |
| Refetch on focus | RQ default |
| Pagination | Cursor-based hooks (`useInfiniteQuery`) for long lists; offset+page for short ones |
| Filters | URL search params (`useSearchParams`) — shareable |

Example: `/schedules` page

```tsx
// app/(app)/schedules/page.tsx (RSC)
export default async function Page({ searchParams }) {
  const queryClient = new QueryClient()
  const { workspaceId } = await searchParams
  await queryClient.prefetchQuery({
    queryKey: ['schedules', workspaceId],
    queryFn: () => fetchSchedules({ workspaceId }),
  })
  return (
    <HydrationBoundary state={dehydrate(queryClient)}>
      <SchedulesScreen workspaceId={workspaceId} />
    </HydrationBoundary>
  )
}
```

```tsx
// components/features/schedules/schedules-screen.tsx ('use client')
export function SchedulesScreen({ workspaceId }) {
  const { data, isLoading } = useSchedules(workspaceId)
  if (isLoading) return <SchedulesSkeleton />
  if (data?.length === 0) return <EmptyState />
  return <SchedulesTable data={data} />
}
```

### 3.2 Form submit (write)

| Step | Mechanism |
|---|---|
| Validate locally | RHF + zodResolver; mirror the server schema |
| Submit | RQ mutation hook; show pending state via `isPending` |
| Success | `onSuccess` → `qc.invalidateQueries(...)` + toast + (optional) navigate |
| Error | `onError` → `setError('root', friendlyError(err))` |

### 3.3 Real-time-ish reads (timeline, notifications)

- v1: RQ polling — `refetchInterval: 30_000` for the timeline; `15_000` for notifications.
- Phase 2: replace with SSE/websockets if polling cost becomes a concern. Keep the hook signature stable.

## 4. Error mapping (backend `code` → user message)

Single source of truth: `~/lib/error-messages.ts`.

```ts
import type { ErrorCode } from '~/server/http/error-response'

const COPY: Record<ErrorCode, string> = {
  UNAUTHENTICATED: 'Sua sessão expirou. Faça login novamente.',
  FORBIDDEN: 'Você não tem permissão para essa ação.',
  NOT_FOUND: 'Não encontramos o que você está buscando.',
  VALIDATION_ERROR: 'Verifique os campos destacados.',
  ALREADY_EXISTS: 'Este recurso já existe.',
  PLAN_LIMIT_REACHED: 'Você atingiu o limite do seu plano. Faça upgrade para continuar.',
  SHIFT_OVERLAP_CONFLICT: 'O usuário já tem um turno conflitante nesse horário.',
  SHIFT_HEADCOUNT_FULL: 'Este turno já está com a equipe completa.',
  DECISION_WINDOW_EXPIRED: 'O prazo para essa decisão expirou.',
  INVALID_STATE_TRANSITION: 'Essa ação não é permitida no estado atual.',
  ALREADY_CLOCKED_IN: 'Você já bateu o ponto de entrada.',
  ATTACHMENT_INVALID: 'O anexo não é válido (formato ou tamanho).',
  ATTACHMENT_UPLOAD_FAILED: 'Não foi possível enviar o anexo. Tente novamente.',
  TIME_OFF_TOO_LARGE: 'A solicitação afeta turnos demais. Divida em períodos menores.',
  EXPORT_TOO_LARGE: 'Exportação muito grande. Filtre por data antes de baixar.',
  // ... (full ErrorCode union)
}

export function friendlyError(err: unknown): string {
  const code = extractCode(err)
  if (code && code in COPY) return COPY[code as ErrorCode]
  return 'Aconteceu um erro inesperado. Tente novamente.'
}
```

**Rules:**
- The map MUST be exhaustive on `ErrorCode` (TypeScript will warn). When the backend adds a code, add a copy entry in the same PR.
- Any backend route returning a new code must update both `error-response.ts` (server) and this file (client).
- Toasts use `friendlyError(err)`. Form root errors do too.

## 5. File uploads (TIME_OFF attachment + future avatar/CSV scenarios)

### 5.1 Frontend

```tsx
// components/features/requests/time-off-form.tsx
const onSubmit = form.handleSubmit(async (values) => {
  const fd = new FormData()
  fd.set('payload', JSON.stringify({
    type: 'TIME_OFF',
    workspaceId: values.workspaceId,
    timeOffStart: values.start,
    timeOffEnd: values.end,
    reason: values.reason,
  }))
  if (values.attachment?.[0]) fd.set('attachment', values.attachment[0])

  await submitRequest.mutateAsync(fd)  // hook calls fetch with multipart
})
```

### 5.2 Hook

```ts
export function useSubmitTimeOff() {
  return useMutation({
    mutationFn: (fd: FormData) =>
      fetch('/api/requests', { method: 'POST', body: fd, credentials: 'include' })
        .then(handleResponse),
  })
}
```

### 5.3 Backend (already shipped — Task 12)

Server parses `formData()`, validates MIME + size, uploads via `~/lib/storage/supabase.ts`, persists `attachmentUrl`. Direct-to-Supabase from the client is **not** allowed (CLAUDE.md storage adapter rule).

## 7. Audit-log → UI events (the read side)

Three frontend surfaces consume audit events:

| Surface | Source | Update strategy |
|---|---|---|
| Shift timeline (F11) | `GET /api/shifts/:id/timeline` projecting from `auditLog` | RQ poll (30s) + `refetchOnWindowFocus` |
| Notifications center (F12) | `GET /api/notifications` (Task 16 — separate `notification` table populated by dispatcher) | RQ poll (15s) |
| Activity log per entity (future, e.g., per-user history) | Same projection layer, different filter | Phase 2 |

The frontend doesn't need to know how the events were generated — it consumes the projected vocabulary documented in [`DESIGN-SYSTEM.md`](DESIGN-SYSTEM.md) §5.7 (Type) and the timeline event types.

## 8. Loading, empty, error states (the boring trinity)

Every list/detail screen ships **all three**:

| State | Pattern | Component |
|---|---|---|
| Loading | Skeleton matching the final layout | `<*Skeleton />` per feature |
| Empty | Illustration + CTA | `<EmptyState />` from `~/components/ui` |
| Error | Friendly message + "Tentar novamente" | Segment-level `error.tsx` |

Skipping any of these is a CLAUDE.md violation ("no half-finished implementations").

---

## 9. Logging + observability (lightweight in v1)

- Browser errors → console (Phase 3 wires Sentry/Axiom — `ROADMAP.md §6.2`).
- Backend already emits `auditLog` for every business action; the frontend doesn't need to duplicate that.
- Add `console.warn` (not `console.error`) for recoverable client-side issues like a stale RQ cache.

## 10. Testing the integration

| Layer | Tool | Example |
|---|---|---|
| Unit (component+hook) | Vitest + RTL | Mock `useShifts` to return canned data; assert renders + interactions |
| Hook isolation | Vitest | Mock `fetch`; assert query key, refetch behavior |
| Browser E2E (critical flow) | Playwright | Real user: login → assign shift → accept → clock-in → clock-out — all on the same DB |
| API regression | Playwright (`tests/api/...`) | Already exists from Phase 1a — keep green |

The "happy path login → assign → accept → clock-in → clock-out → close" is the canonical smoke test for every release. It exercises ~70% of the system.

## 11. Versioning + breaking changes

The backend is unversioned in v1 (single `/api/...` path). Phase 1b consumes whatever the backend currently exposes.

If a backend change is breaking:

1. Update the backend with a new route or new field; keep the old surface returning the old shape for a release.
2. Update the frontend hook + components.
3. Remove the old surface in a follow-up PR once analytics confirm zero traffic.

This applies equally to GraphQL — additive changes always (new fields, new arguments). Removing a field requires the dual-write window above.

## 12. Where to start a new feature

1. Read the relevant Phase 1a brief in `docs/tasks/<api-slug>/brief.md` to know the contract.
2. Open the corresponding row in [`FRONTEND-ROADMAP.md`](FRONTEND-ROADMAP.md) §4.
3. Create `docs/tasks/<frontend-slug>/brief.md` from `docs/_templates/task-brief-template.md`.
4. Run the 5-phase workflow (CLAUDE.md).
