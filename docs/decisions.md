## 2026-07-22

### D-011 · Marca/modelo de veículo: catálogo real importado da FIPE uma vez, não texto livre
**Contexto:** durante o QA manual do S10-T1, o usuário apontou que os campos `brand`/`model` do `VehicleForm` (texto livre) deixariam o carrier digitar o mesmo fabricante de formas diferentes ("Fiat" vs "FIAT" vs "fiat"). Pesquisa confirmou a Tabela FIPE (`https://fipe.parallelum.com.br/api/v2`) como catálogo público oficial brasileiro, com 3 tipos (`cars`/`motorcycles`/`trucks`, mapeando 1:1 nas categorias já criadas — Van/Moto/Caminhão), cada um com marca → modelo hierárquico (107/102/29 marcas respectivamente). Rate limit da API é real (~500-1000 req/dia compartilhado) — não permite chamada ao vivo por cadastro. **Decisão (Good):** dois models novos, `VehicleBrand`/`VehicleModel`, ortogonais a `VehicleCategory`/`VehicleSpec` (marca/modelo = identidade do veículo; categoria/spec = capacidade de carga) — `VehicleCategory` ganhou um campo `fipeVehicleType` só pra filtrar `VehicleBrand` por categoria, sem FK direta. Import único via script novo (`prisma/seed/vehicle-brands-models.ts`, idempotente/retomável — pula marcas já importadas, com retry+backoff em 429), não parte do `db:seed` padrão (mesmo padrão de `dev-public-search-fixture.ts`: reference data pesada roda manualmente). `Vehicle.brand`/`Vehicle.model` (texto livre) viraram `Vehicle.modelId` (FK). `VehicleForm` ganhou uma segunda cascata (Marca → Modelo) ao lado da já existente (Categoria → Especificação). **Alternativas descartadas:** lista curada à mão (catálogo incompleto, manutenção manual) e manter texto livre só normalizado (não resolve o problema real). **Motivo:** escolha explícita do usuário via Fast/Good/Ideal em chat — catálogo real e completo sem depender de chamada externa em runtime. **Nota operacional:** o import bateu rate limit da FIPE após ~15 marcas de moto (de 238 no total); decisão do usuário foi pausar e seguir com o resto do S10-T1 usando o catálogo parcial (Moto com ~15 marcas reais, Van/Caminhão só com uma marca "Fixture" de teste) — completar o import fica de follow-up, rodando o script novamente (idempotente) fora do rate limit.

**Incidente registrado nesta mesma rodada:** ao tentar resolver um bloqueio de push do Prisma (coluna obrigatória nova em tabela com linhas existentes), rodei `TRUNCATE TABLE "vehicleCategory" CASCADE` esperando limpar só a tabela pequena (3 linhas) e suas dependentes diretas (`vehicleSpec`, `vehicle`) — `TRUNCATE ... CASCADE` do Postgres ignora `onDelete: SetNull` da FK e trunca fisicamente qualquer tabela referenciando a truncada, o que apagou também `shipment`/`proposal`/`review`/`chatMessage`/etc. inteiros (todo o histórico de QA da sessão, não só dado de veículo). Sem dado de produção real perdido (ambiente pré-lançamento), usuário optou por seguir e reseedar. Lição registrada: `TRUNCATE ... CASCADE` tem blast radius diferente de `DELETE` com as mesmas FKs — não usar como atalho pra "limpar uma tabela pequena" sem checar toda a árvore de dependência primeiro.

### D-010 · Sprint 10 (Frota & Match): escopo em duas tasks sequenciais, taxonomia hierárquica como direção do modelo de veículo
**Contexto:** usuário pediu pra aprofundar dados de veículo/frota e o algoritmo de match, apontando `build-track-api` (repo irmão, marketplace de serviços domésticos) como referência estrutural. Investigação confirmou dois gaps reais: `VehicleType` é um enum achatado de 5 valores sem capacidade (peso/volume), e cada `Vehicle` só tem 1 tipo — sem representar frota mista; e o "match" hoje é fila FIFO pura (`join-proposal-queue.use-case.ts`/`refill-called-group.ts`), sem checar veículo, distância ou nota — `Shipment.vehicleTypeRequired` é gravado mas nunca lido em lugar nenhum do fluxo de fila/proposta. Descoberta adicional durante o Brief: **não existe nenhum CRUD de veículo hoje** — `VehicleRepository` só tem leitura (`findActiveTypeByOwnerId`), as únicas linhas de `Vehicle` no banco vêm de um seed de dev, não de um fluxo real de cadastro. build-track não tem domínio de veículo (é serviços domésticos), mas tem o padrão estrutural transferível: `SpecialtyGroup → SpecialtyItem` (taxonomia hierárquica, many-to-many com o que o job exige) e um dispatch automático (`findBestAutonomous`) que filtra por nota mínima + overlap de especialidade + raio geográfico, ordena por distância/nota e pega top-5 — mas o próprio ROADMAP do build-track *descreve* um score ponderado (rating/distância/tempo de resposta/histórico) que **não está implementado**; o código real é só filtro + ordenação simples, não um score composto de verdade. **Decisão:** duas tasks sequenciais no ROADMAP (Sprint 10) — **S10-T1** (taxonomia de frota: categoria → especificação com capacidade real, CRUD de veículo pro carrier, migração dos 4 pontos que hoje leem o enum antigo) antes de **S10-T2** (algoritmo de match: elegibilidade por frota + outros filtros — distância/raio, nota, disponibilidade — no fluxo de fila/proposta), porque o match depende do modelo de dados existir primeiro. Modelo de veículo = taxonomia hierárquica (não só enriquecer o enum, não só capacidade contínua sem categoria), desenhada desde já pra suportar múltiplos filtros no S10-T2, não só compatibilidade de veículo. **Motivo:** escolha explícita do usuário via Fast/Good/Ideal em chat — recomendação (tier "Good"/"taxonomia hierárquica") aceita nos dois casos, com a ressalva de que o desenho deve antecipar o leque maior de filtros do match futuro.

### D-009 · Toasts de erro: retrofit dos hooks GraphQL sem mapeamento de código, em vez de reescrever tudo num utilitário central
**Contexto:** usuário reportou toasts de erro mostrando mensagem "de dev" em vez de amigável. Investigação encontrou a causa raiz em `providers/query-provider.tsx` — o fallback global de `onError` fazia `toast.error(`Erro...: ${error.message}`)`, e `ClientError.message` (graphql-request) despeja o JSON cru da requisição/resposta GraphQL, não uma mensagem exibível (`getGraphQLErrorCode`, já existente em `lib/graphql-client.ts`, resolve isso mas só era usado em 9 dos ~38 hooks GraphQL). Também achado: `login-form.tsx`/`register-form.tsx` tinham lógica invertida — só mostravam a tradução PT-BR em erro de rede, e a string em inglês crua do servidor no caminho esperado (credenciais erradas, e-mail em uso). Pergunta levantada em chat (Fast/Good/Ideal): escopo do fix — só o fallback global, ou também retrofitar os hooks sem mapeamento e os bugs de login/registro, ou ainda consolidar tudo num utilitário central. **Decisão (Good):** (1) `query-provider.tsx` agora usa `getGraphQLErrorCode` — se o erro tem um código de extensão GraphQL, mostra uma mensagem genérica amigável em vez do JSON cru; (2) os 9 hooks de mutation sem mapeamento (`use-accept-proposal`, `use-reject-proposal`, `use-confirm-delivery`, `use-confirm-safety-check-in`, `use-mark-collected/in-transit/delivered`, `use-publish-shipment`, `use-submit-review`) ganharam o mesmo padrão `ERROR_MESSAGES` + `xxxErrorMessage()` já usado em `use-submit-proposal`/`use-create-shipment`; (3) 4 hooks de query cujo componente já renderiza `EmptyState` próprio (`use-my-shipments`, `use-browse-shipments`, `use-my-proposals`, `use-carrier-documents`) ganharam `meta: { silent: true }`, mesmo padrão já usado em `use-shipment.ts`; (4) `login-form.tsx`/`register-form.tsx` corrigidos pra nunca mostrar `err.message` cru — login sempre mostra a mensagem PT-BR fixa, registro mapeia por `err.status` (409 → e-mail em uso, resto → erro genérico). **Não foi feita** a consolidação num `src/lib/error-messages.ts` central (tier "Ideal") — os ~20 mapas locais (`errorCopy`/`ERROR_MESSAGES`/`errorMap`) que já existiam continuam duplicados. **Motivo:** escolha do usuário — resolver o vazamento real com mensagens específicas por ação (consistente com o padrão já estabelecido em 9 hooks) sem arriscar tocar código que já funciona numa refatoração maior.

### D-008 · Histórico do frete: frase amigável em PT-BR, mesma lista pros dois lados (customer e carrier)
**Contexto:** review de referências do Metrobi (concorrente, screenshots em `/Users/davidlucas/Downloads/metrobi-references`) mostrou um timeline de eventos por frete. O backend já tinha tudo pronto e sem uso no GraphQL: `enum EventType` (11 valores), `model ShipmentEvent`, e `shipmentEventRepo.listByShipment()` (só consumido por uma rota REST, `getShipmentEvents` use-case). Pergunta levantada em chat (Fast/Good/Ideal): granularidade e simetria entre papéis. **Decisão (Good):** um mapa fixo `EventType → frase PT-BR` (`server/graphql/shipment-event-descriptions.ts`), exposto por uma única query `shipmentEvents(shipmentId)`, retornando exatamente a mesma lista/texto pra `CUSTOMER` e `CARRIER` — sem resolução de nome de ator, sem ícone por tipo de evento, sem filtro por papel (isso ficou pro tier "Ideal", não escolhido). Autorização reaproveita `resolveSafetyParticipant`, mesmo padrão já usado por `safetyCheckInStatus`/`deliveryConfirmationStatus`/`reviewsForShipment`. **Motivo:** escolha do usuário — menor esforço com resultado correto; a diferenciação por papel/ator pode ser adicionada depois se algum caso de uso real pedir.

### D-007 · Cartão de contato da contraparte: nome + nota sempre visíveis, telefone atrás de um clique
**Contexto:** mesmo review Metrobi — a referência mostra um cartão de contato do motorista (nome, nota, telefone, foto) assim que uma entrega é atribuída. No Movux hoje (após `CARRIER_SELECTED`), o customer nunca vê a identidade do transportador (nem o carrier vê a do customer) — só status/preço. `CarrierProfile.phone` é obrigatório no schema; `CustomerProfile.phone` é opcional. Pergunta levantada em chat (Fast/Good/Ideal): quanto expor e quando. **Decisão (Good):** uma única query `shipmentCounterpartInfo(shipmentId)` que resolve a contraparte certa a partir do papel autenticado (`CUSTOMER` recebe dados do carrier aceito; `CARRIER` recebe dados do customer dono do frete), disponível a partir de `CARRIER_SELECTED` em diante. Nome e nota (`avgRating`) vêm sempre preenchidos na resposta; o telefone também vem na mesma query (sem mascaramento no backend, já que a query é escopada ao participante autenticado do frete) mas só é revelado na UI depois de um clique explícito em "Mostrar telefone" (`ShipmentCounterpartCard`, montado nos dois lados — `shipment-detail-view.tsx` e `carrier-shipment-detail-view.tsx`). **Motivo:** escolha do usuário — simetria entre os dois papéis (uma query, um componente reaproveitado) em vez de construir só o lado customer→carrier; a fricção do clique reduz exposição acidental do telefone sem exigir infraestrutura de mascaramento no backend.

### D-006 · Domínio Turnora órfão (tenant/workspace/shift/time-entry/etc.) fica como está — não remover nem restaurar schema agora
**Contexto:** durante a QA do Sprint 9, `pnpm lint`/`pnpm build` completos falharam com ~249 erros TS em 56 arquivos (`server/repositories/{tenant,workspace,shift,time-entry,transfer-request,user-specialty,...}.repository.ts`, use-cases e rotas correspondentes) — todos referenciando models Prisma (`Tenant`, `Workspace`, `Shift`, `TimeEntry`, `TransferRequest`, `UserSpecialty` etc.) que **não existem mais** em `schema.prisma` desde a adoção do domínio Movux (D-002). A migração de domínio trocou o schema mas nunca removeu o código de repository/use-case/rota do domínio antigo — débito já mencionado nos Follow-ups de `validation.md` de S8-T1/T4/T5/T6, mas nunca promovido a decisão explícita. **Decisão:** manter como está por enquanto — não apagar os 56 arquivos nem restaurar os models antigos no schema. Qualquer task nova documenta esse débito como known issue em vez de tentar corrigi-lo de passagem; uma remoção completa do domínio órfão vira uma task própria (Brief→Exploration→Research→Plan) se/quando for priorizada. **Motivo:** decisão explícita do usuário ao ser confrontado com o escopo real (56 arquivos, não 1 linha) — remover ou restaurar é trabalho grande o suficiente pra merecer o processo completo, não uma correção incidental durante QA de outra sprint.

### D-005 · Navegação desktop: sidebar lateral fixa, sempre — nunca nav horizontal no topo
**Contexto:** durante o levantamento de referências visuais pro redesign (ver [`docs/design-references-notes.md`](design-references-notes.md)), duas referências mostraram padrões de navegação desktop incompatíveis: `Desktop.svg` usa sidebar lateral fixa (~280px); as telas do "Financy" (Dashboard/Categorias/Transações) usam nav horizontal no topo (logo à esquerda, links centralizados, avatar à direita). O Movux já tem sidebar implementada (`components/features/nav/sidebar.tsx` + `app-shell.tsx`, `md:flex`, `w-64`). **Decisão:** sidebar lateral fixa é definitiva para desktop — nenhuma referência futura de nav horizontal deve ser considerada pra estrutura de navegação; referências futuras devem ser lidas só pela experiência de corpo/conteúdo (cards, listas, forms, modais), não pela chrome de navegação. **Motivo:** decisão explícita do usuário; evita retrabalho estrutural a cada nova referência mandada.

## 2026-07-20

### D-004 · Sprint 8 (UI) consome GraphQL, não REST — apesar da API REST já ter 108 rotas testadas
**Contexto:** ao escopar o S8-T1 (primeira tela real do Sprint 8, fluxo de fretes do customer), havia duas opções de data layer: REST via `api-client` (padrão usado pelos Sprints 0-6, zero trabalho de backend novo) ou GraphQL (schema Pothos hoje só tem `tenant`/`me`, domínio Turnora — nada de shipment/proposal). **Decisão:** GraphQL, usando o stack já documentado no CLAUDE.md (`graphql-request` + React Query, **sem Axios** — a regra "no Axios" do CLAUDE.md continua valendo). Isso significa que cada task de UI do Sprint 8 que expõe um domínio novo (shipment, proposal, carrier, review) precisa construir as queries/mutations Pothos correspondentes antes das telas. **Motivo:** escolha explícita do usuário, priorizando cache/performance de query (React Query sobre GraphQL) sobre reaproveitar a REST já pronta — aceito o trade-off de escopo maior por task. Full plan: [`docs/tasks/s8-t1-customer-shipments-ui/`](tasks/s8-t1-customer-shipments-ui/).

### D-002 · Estrutura técnica baseada no Turnora; filosofia de produto baseada no Build-track
**Contexto:** o usuário possui dois repos de referência: `turnora` (monorepo Next.js App Router + GraphQL/Pothos + CASL RBAC + clean architecture) e `build-track` (marketplace bilateral de serviços: autônomo ↔ cliente, proposta, seleção, status de serviço). **Decisão:** clonar o Turnora como base técnica (stack, estrutura de pastas, guardrails, padrões de código) e usar o Build-track como referência de filosofia de produto (fluxo de marketplace, entidades, lifecycle). Domínio do Turnora (turnos, escalas, hospitais) removido; substituído pelo domínio Movux (fretes, mudanças, segurança progressiva). **Motivo:** aproveitar infraestrutura madura sem reescrever do zero; separar a escolha técnica da escolha de produto.

### D-001 · Nome do produto: Movux
**Contexto:** brainstorm de nomes para marketplace de fretes e mudanças. Candidatos avaliados: Levo, Levux, FastFee, Rutex, Bora, Corre, Ferrox, Knox, Junto. Critérios: vira verbo naturalmente em PT-BR, curto, sem acento, soa tech/startup, memorável. **Decisão:** **Movux** — *mover* + *ux* (movimento + experiência). Vira verbo: *"chama um Movux"*. Internacional sem perder o sentido. Visualmente forte como marca. **Motivo:** melhor equilíbrio entre os critérios; preferência do usuário após avaliação comparativa.
